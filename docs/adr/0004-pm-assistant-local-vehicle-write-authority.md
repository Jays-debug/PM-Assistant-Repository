# ADR-0004: PM Assistant Local Vehicle Write Authority

- Status: Accepted
- Date: 2026-07-21
- Decision owner: FleetOS Product Owner
- Scope: Phase 6.3 — PM Assistant-local Vehicle reference creation authority

## Context

Phases 5.2 through 6.1 established an immutable Original Vehicle Number, a
minimal PM Assistant-local Vehicle Aggregate, and a read-only vertical slice.
Those approvals did not authorize Vehicle creation, mutation, deletion, or
enterprise Vehicle Master ownership.

PM Assistant needs an explicit decision before a later phase can design its
first local Vehicle write use case. The decision must preserve the distinction
between PM Assistant-local entity continuity and future enterprise Vehicle
identity.

## Decision

1. PM Assistant is authorized to create a Vehicle reference for its own local
   maintenance use.
2. This local authority does not make PM Assistant the owner of an enterprise
   Vehicle Master and does not authorize `fleetos_vehicle_id`.
3. `local_vehicle_id` is generated by persistence inside the application-owned
   transaction. The aggregate does not generate it.
4. Callers may not supply or select `local_vehicle_id` for creation.
5. A generated local identifier remains an immutable positive integer scoped
   only to PM Assistant. It is not derived from Original Vehicle Number, is not
   a public or cross-system identity, and need not be contiguous.
6. Successful local Vehicle creation must be auditable. The required audit
   content, persistence shape, access, retention, and implementation remain
   deferred.
7. One local creation command owns one explicit unit-of-work boundary. Required
   state and required audit evidence must satisfy their later approved
   consistency contract before success is reported.
8. Repository or adapter operations do not commit independently. A known
   failure rolls back the application-owned scope.
9. Automatic retry is prohibited when a repeated attempt could create a second
   accepted outcome. An uncertain commit outcome is not success and requires
   reconciliation before retry.

## Pending Original Vehicle Number uniqueness policy

Whether exact Original Vehicle Number values must be unique among PM
Assistant-local Vehicle references remains a pending Product Owner decision.
The current database constraint is implementation evidence only and does not
establish an architectural invariant.

Until the policy is approved:

- an application write contract must not invent duplicate semantics;
- exact value equality must not be treated as proof of Vehicle identity;
- normalization, case folding, trimming, matching, and alias behavior remain
  excluded;
- a persistence adapter must not translate a storage uniqueness failure into a
  domain duplicate conflict without approved authority.

## Explicit deferrals

This ADR does not authorize:

- updating Original Vehicle Number or any other Vehicle state;
- deletion, retirement, archive, tombstone, merge, or split;
- enterprise Vehicle Master ownership or lifecycle;
- `fleetos_vehicle_id` or another global identifier;
- normalization, matching, aliases, grouping, or reconciliation;
- a generic repository, command bus, event bus, or domain events;
- a write API, authentication, authorization, or public write-error contract;
- a database migration, schema choice, audit structure, or storage technology;
- AutoPM writes or AutoPM integration.

## Transaction and failure direction

- The application service owns the creation transaction.
- Persistence may allocate the local identifier while staging the new record,
  but the identifier is not a successful result until commit is confirmed.
- Validation, staging, auditability, commit, rollback, closure, and uncertain
  outcomes follow the FleetOS transaction model.
- Rolled-back allocation may leave an identifier gap; gaps and ordering have no
  domain meaning.
- Correlation IDs, timestamps, local identifiers, and Original Vehicle Numbers
  are not automatically idempotency keys.

## Consequences

### Positive

- A later phase may design one narrow PM Assistant-local creation use case.
- Local identity generation has one owner and cannot be selected by callers.
- Enterprise identity and module boundaries remain unchanged.
- Transaction ownership and uncertain outcomes are explicit before code work.

### Negative

- Creation implementation remains blocked until duplicate policy and minimum
  audit compatibility are sufficiently resolved for the selected persistence
  design.
- Persistence-generated identity requires careful staging so an allocated ID
  cannot escape as a successful result before commit.
- Existing legacy write behavior cannot be reused as architecture authority.

## Risks and mitigations

| Risk | Mitigation |
| --- | --- |
| Local creation is mistaken for enterprise Vehicle Master authority | Keep the scope PM Assistant-local and retain the enterprise ownership decision as unresolved. |
| A caller selects or predicts identity | Prohibit caller-supplied IDs and assign no meaning to sequence or gaps. |
| Current database uniqueness becomes business policy accidentally | Keep Original Vehicle Number uniqueness pending and treat the current constraint as evidence only. |
| Creation is reported before durability | Return success only after confirmed application-owned commit. |
| An uncertain outcome is replayed blindly | Require reconciliation before retry and never report uncertain commit as success. |
| Audit design is invented prematurely | Require auditability while deferring fields, shape, storage, access, and retention. |

## Alternatives considered

### Caller-supplied local identity

Rejected because it transfers persistence identity allocation to an untrusted
boundary and risks treating a local integer as a business or public identifier.

### Aggregate-generated local identity

Rejected for this scope because the approved aggregate represents a positive
integer identity and the existing direction does not authorize a domain ID
generator.

### Treat current exact database uniqueness as approved policy

Not accepted. Storage constraints cannot resolve the pending business decision
about duplicate Original Vehicle Numbers.

### Authorize full Vehicle lifecycle

Rejected. Update, correction, reuse, retirement, deletion, merge, split, and
enterprise identity require separate decisions.

## Rollback and supersession

This documentation-only decision changes no runtime state or data. Before
implementation, its documentation can be reverted as one isolated set. After
acceptance, a policy reversal should use a superseding ADR so the original
authority and rationale remain traceable.

## Result

Later phases may plan a narrow PM Assistant-local Vehicle creation use case only
within this authority. Write implementation remains blocked until separately
approved scope resolves the pending duplicate policy and demonstrates the
required transactional auditability.
